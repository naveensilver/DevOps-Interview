“In our pipelines, we integrated security scanning in a way that balances speed and security by shifting critical checks left and optimizing when and how scans run.

* Critical scans → run in real-time (blocking)
* Deep scans (like full vulnerability scans or compliance checks) → run asynchronously in parallel or scheduled jobs

Fifth, we parallelized stages in Jenkins pipelines so scanning does not block other independent steps.

For example:

* Build and scan run in parallel branches
* Deployment waits only for required checks

Finally, we defined thresholds — only high/critical vulnerabilities fail the pipeline, while medium/low are reported but don’t block deployment.

This approach ensures strong security enforcement without significantly impacting deployment speed.”

## 🎤 What is Jenkins Shared Library? (Gold Format Answer)

### ✅ What
A Jenkins Shared Library is a **centralized place to write reusable pipeline code** that can be shared across multiple Jenkins jobs or microservices.

Instead of writing the same pipeline logic in every Jenkinsfile, we define common functions once and reuse them everywhere.

---

### 🎯 Why
We use Shared Libraries because:

- Avoid duplicate pipeline code across services  
- Make pipelines easier to maintain and update  
- Ensure **standardized CI/CD flow** across all applications  
- Reduce human errors from copy-paste Jenkinsfiles  
- Improve scalability when number of microservices increases  

In simple terms — *write once, use everywhere.*

---

### ⚙️ How it works
- We create a Git repository for the shared library
- Inside it, we define reusable functions (Groovy scripts)
- In Jenkinsfile, we just call these functions instead of writing full logic
- Jenkins loads this library at runtime and executes it

Example structure:
- `vars/` → reusable pipeline steps (build, deploy, test)
- `src/` → helper logic (if needed)

---

### 💡 Real Example (My Project Use Case)

In my project, we had around 6–8 microservices, and each service had its own Jenkinsfile.

Earlier:
- Every Jenkinsfile had repeated logic for build, test, Docker image creation, and deployment
- If we had to change something, we had to update all pipelines manually

---

### 🛠️ What I did:

I created a **Jenkins Shared Library** and moved common stages like:
- build application
- run unit tests
- build Docker image
- deploy to Kubernetes

into reusable functions.

Then each microservice Jenkinsfile became very simple:

```groovy
@Library('my-shared-lib') _

pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                buildApp()
            }
        }
        stage('Test') {
            steps {
                runTests()
            }
        }
        stage('Deploy') {
            steps {
                deployToK8s()
            }
        }
    }
}
```

## 🎤 How do Kubernetes agents work in Jenkins?

### ✅ What
Kubernetes agents in Jenkins are **dynamic build environments where each Jenkins job runs inside a Kubernetes pod instead of a static server**.

So instead of using fixed Jenkins nodes, Jenkins creates a temporary pod for each build and destroys it after the job is completed.

---

### 🎯 Why
We use Kubernetes agents because:

- No need to maintain static Jenkins servers
- Each build gets a **clean isolated environment**
- Better scalability during high number of builds
- No dependency conflicts between jobs
- Efficient resource usage (pods are created only when needed)

In simple terms — *each build gets its own temporary container in Kubernetes.*

---

### ⚙️ How it works
1. Developer triggers a Jenkins job
2. Jenkins talks to Kubernetes cluster
3. Kubernetes creates a **pod dynamically**
4. That pod acts as a Jenkins agent
5. Build runs inside the pod (checkout, build, test, docker, etc.)
6. Once build is done, pod is destroyed automatically

So lifecycle is:
👉 Create Pod → Run Job → Destroy Pod

---

### 💡 Real Example (My Project Use Case)

Earlier in my project:
- We were using static Jenkins agents
- They had fixed capacity and sometimes got overloaded
- We also faced dependency conflicts between builds

---

### 🛠️ What I improved:

I migrated Jenkins agents to **Kubernetes-based dynamic pods**.

So for every pipeline:
- A fresh pod was created
- It had required tools like Maven, Docker, Node.js based on job requirement
- After build completion, the pod was removed automatically

---

### 📊 Result
- No more agent contention issues
- Faster build scheduling
- Better resource utilization in cluster
- Easy scaling during peak build times
- Clean environments for every build (no leftover dependencies)

---

### 💡 Final takeaway
Kubernetes agents made our Jenkins setup **cloud-native, scalable, and maintenance-free**, because we no longer depended on static servers and each build became fully isolated and reproducible.
