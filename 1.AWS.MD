<details>
<summary><b>Explain how you design a highly available system in AWS</b></summary>

**What**

Designing a highly available system in AWS means building an architecture that **has no single point of failure and can automatically handle failures across AZs or components**.

**Why**

We design HA systems to:

* Avoid downtime
* Handle traffic spikes
* Ensure reliability during failures (node / AZ / infra)

**How (Interview Flow – Easy to Remember)**

1. Global Layer (Traffic Routing)

* Use **Route53**
* Routes users to the application endpoint

✔ Entry point for all traffic

2. Network Layer (VPC Design)

* Create **VPC with public and private subnets across 2–3 AZs**
* Public subnets → Load Balancer
* Private subnets → Application (EKS nodes)

✔ Ensures isolation + HA across AZs

3. Availability Layer (Multi-AZ – MUST)

* Deploy all components across **multiple AZs**
* Example:

  * ap-south-1a, ap-south-1b, ap-south-1c 
  * eu-central-1a, eu-central-1b, eu-central-1c (Germany → eu-central-1)
  * Canada → ca-central-1

✔ Handles AZ failure

4. Load Balancing Layer

* Use **Application Load Balancer (ALB)**
* Placed in public subnets
* Distributes traffic across:

  * multiple AZs
  * multiple application instances/pods 

✔ Ensures traffic distribution + fault tolerance

5. Compute Layer (EKS + Auto Scaling)

* Deploy application on **EKS**
* Worker nodes distributed across AZs
* Configure:

  * **HPA** → pod scaling
  * **Cluster Autoscaler** → node scaling

✔ Handles traffic spikes + scaling

6. Self-Healing Layer

* Kubernetes ensures:

  * Pod restarts (if failure)
  * Rescheduling on healthy nodes

* Use:

  * **Liveness Probe**
  * **Readiness Probe**

✔ Ensures only healthy pods serve traffic

7. Database Layer

* Use **RDS Multi-AZ**
* Primary + standby replica in another AZ
* Automatic failover using same endpoint

✔ Eliminates DB single point of failure

8. Storage Layer

* Use **S3 for static content**
* Automatically replicated across AZs

✔ High durability + availability

📊 9. Monitoring & Health Checks

* Use:

  * **CloudWatch** → infra monitoring
  * Kubernetes metrics → pod health

✔ Detect issues early

<summary><b>Real Example (Final Interview Answer)</b></summary>

<br>

“To design a highly available web application across multiple Availability Zones, I ensure there is no single point of failure at any layer.

At the entry level, I use Route53 to route user traffic to the application.

In the network layer, I create a VPC with public and private subnets across at least two or three Availability Zones.

I deploy an Application Load Balancer in public subnets, which distributes incoming traffic across application services running in private subnets across multiple AZs.

For the application layer, I use Amazon EKS with worker nodes distributed across multiple AZs.
I configure multiple pod replicas along with HPA for pod scaling and Cluster Autoscaler for node scaling.

Kubernetes ensures self-healing using liveness and readiness probes, and ALB performs health checks to route traffic only to healthy instances.

For the database layer, I use Amazon RDS with Multi-AZ enabled, which provides automatic failover using a standby replica without changing application endpoints.

For static storage, I use Amazon S3, which is inherently highly available across AZs.

For monitoring, I use CloudWatch and Kubernetes metrics to track application and infrastructure health.

The overall flow is:

**User → Route53 → ALB → EKS (Multi-AZ Pods) → RDS (Multi-AZ) → Monitoring**

This ensures high availability, scalability, and automatic failover even if an Availability Zone goes down.”

AWS provides different storage services to store **files, block data, and shared data** based on use case.

<summary><b>S3 Bucket Policy vs IAM Role — what is the difference?</b></summary>

### 🔥 Perfect Answer (Say This)

<br>

“IAM roles and S3 bucket policies are both used to control access, but they work at different levels.

IAM roles are identity-based policies, which define what actions a user, service, or resource can perform on AWS resources.

S3 bucket policies are resource-based policies, which define who can access a specific S3 bucket and what actions they can perform.

In simple terms, IAM controls ‘who can do what’, and bucket policies control ‘who can access this resource’.

In my project, we used IAM roles for applications running on EKS to access S3 securely using IRSA, and we used bucket policies to restrict access to specific roles and block public access.

So both work together to provide secure and fine-grained access control.”

| ----------- | -------------------- | --------------------- |
| Type        | Identity-based       | Resource-based        |
| Attached to | User/Role/Service    | S3 Bucket             |
| Controls    | What actions allowed | Who can access bucket |
| Scope       | Multiple services    | Only S3               |

<summary><b>EC2 is running, but app is not accessible via domain. Where exactly will you check across Route53, ALB, Security Groups, NACLs?</b></summary>

<br>

#### 🔥 Perfect Answer (Production-Level)

“When EC2 is running but the application is not accessible via domain, I debug layer by layer starting from DNS to network.

First, I check Route53. I verify whether the domain is correctly pointing to the ALB using an A record (Alias). I also check if DNS is resolving properly using tools like nslookup or dig.

Next, I move to the ALB. I check:

* Whether the ALB listener is configured on the correct port (80/443)
* If the listener rules are correctly forwarding traffic to the target group
* Target group health — if targets are unhealthy, traffic won’t reach EC2

Then, I check the target group:

* Whether the EC2 instance is properly registered
* Health check path and port configuration
* If health checks are failing, I verify whether the application endpoint is responding

After that, I check Security Groups:

* ALB security group → should allow inbound traffic from internet (0.0.0.0/0) on required ports
* EC2 security group → should allow inbound traffic from ALB security group on application port

Then, I verify NACLs:

* Ensure inbound and outbound rules allow traffic for required ports (HTTP/HTTPS + ephemeral ports)
* Confirm no deny rules blocking traffic

I also check on EC2:

* Application is running and listening on the correct port
* Port is open (netstat / ss)
* Application logs for any runtime errors

Finally, I validate end-to-end connectivity by hitting the ALB DNS directly and then the domain, to isolate whether the issue is DNS or backend related.

This structured approach helps quickly pinpoint whether the issue is DNS, load balancer, or network misconfiguration.”

---

#### 🧠 Simple Way to Remember

👉 5-step debugging flow:

1. Route53 (DNS)

* Correct record (A/ALIAS)
* DNS resolving

2. ALB

* Listener (80/443)
* Listener rules

3. Target Group

* Instance registered
* Health check passing

4. Security Groups

* ALB → internet access
* EC2 → allow from ALB

5. NACLs + EC2

* Allow ports + ephemeral
* App running on correct port

## ECS VS EKS 

<details>
<summary><b>What is ECS vs EKS and when do you use which?</b></summary>

<br>

## 🔹 What is ECS?

**Amazon ECS (Elastic Container Service)** is a fully managed container orchestration service by AWS.

👉 Key idea:

* AWS manages orchestration for you
* You just run containers

👉 You can run ECS in two modes:

* **EC2 mode** → you manage servers
* **Fargate mode** → serverless (no infra management)

---

## 🔹 What is EKS?

**Amazon EKS (Elastic Kubernetes Service)** is a managed Kubernetes service.

👉 Key idea:

* AWS manages control plane
* You manage Kubernetes workloads

👉 You get:

* Full Kubernetes ecosystem
* kubectl, Helm, CRDs, etc.

---

## 🔥 Core Difference (Don’t Miss This)

| Aspect         | ECS               | EKS                  |
| -------------- | ----------------- | -------------------- |
| Complexity     | Simple            | Complex              |
| Learning curve | Low               | High                 |
| Flexibility    | Limited           | Very high            |
| Ecosystem      | AWS-native        | Kubernetes ecosystem |
| Control        | Less              | More                 |
| Portability    | Low (AWS lock-in) | High (multi-cloud)   |


#### 🔥 Perfect Answer (Production-Level)

“ECS and EKS are both container orchestration services in AWS, but they differ in complexity, flexibility, and use cases.

ECS is AWS’s native container service, which is simpler to set up and manage. It’s tightly integrated with AWS services and is ideal when you want to quickly deploy containers without dealing with Kubernetes complexity.

EKS, on the other hand, is a managed Kubernetes service. It provides full Kubernetes capabilities, which gives more flexibility, scalability, and portability across environments.

In terms of usage, I choose ECS when:

* The application is simple or microservices are not very complex
* The team doesn’t have Kubernetes expertise
* Faster setup and lower operational overhead is required

I choose EKS when:

* I need advanced orchestration features like auto-scaling, service mesh, or custom controllers
* There is a need for multi-cloud or vendor-neutral architecture
* The system is complex with multiple microservices requiring fine-grained control

In my experience, for enterprise-grade, scalable microservices platforms, EKS is preferred despite its complexity, while ECS is better for simpler, faster deployments.”

---

#### 🧠 Simple Way to Remember

👉 Decision rule:

* **ECS = Simplicity + Speed**
* **EKS = Control + Flexibility**

---
