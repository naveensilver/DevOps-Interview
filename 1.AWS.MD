<details>
<summary><b>Explain how you design a highly available system in AWS</b></summary>

**What**

Designing a highly available system in AWS means building an architecture that **has no single point of failure and can automatically handle failures across AZs or components**.

**Why**

We design HA systems to:

* Avoid downtime
* Handle traffic spikes
* Ensure reliability during failures (node / AZ / infra)

**How (Interview Flow – Easy to Remember)**

1. Global Layer (Traffic Routing)

* Use **Route53**
* Routes users to the application endpoint

✔ Entry point for all traffic

2. Network Layer (VPC Design)

* Create **VPC with public and private subnets across 2–3 AZs**
* Public subnets → Load Balancer
* Private subnets → Application (EKS nodes)

✔ Ensures isolation + HA across AZs

3. Availability Layer (Multi-AZ – MUST)

* Deploy all components across **multiple AZs**
* Example:

  * ap-south-1a, ap-south-1b, ap-south-1c 
  * eu-central-1a, eu-central-1b, eu-central-1c (Germany → eu-central-1)
  * Canada → ca-central-1

✔ Handles AZ failure

4. Load Balancing Layer

* Use **Application Load Balancer (ALB)**
* Placed in public subnets
* Distributes traffic across:

  * multiple AZs
  * multiple application instances/pods 

✔ Ensures traffic distribution + fault tolerance

5. Compute Layer (EKS + Auto Scaling)

* Deploy application on **EKS**
* Worker nodes distributed across AZs
* Configure:

  * **HPA** → pod scaling
  * **Cluster Autoscaler** → node scaling

✔ Handles traffic spikes + scaling

6. Self-Healing Layer

* Kubernetes ensures:

  * Pod restarts (if failure)
  * Rescheduling on healthy nodes

* Use:

  * **Liveness Probe**
  * **Readiness Probe**

✔ Ensures only healthy pods serve traffic

7. Database Layer

* Use **RDS Multi-AZ**
* Primary + standby replica in another AZ
* Automatic failover using same endpoint

✔ Eliminates DB single point of failure

8. Storage Layer

* Use **S3 for static content**
* Automatically replicated across AZs

✔ High durability + availability

📊 9. Monitoring & Health Checks

* Use:

  * **CloudWatch** → infra monitoring
  * Kubernetes metrics → pod health

✔ Detect issues early

<details>
<summary><b>Real Example (Final Interview Answer)</b></summary>

<br>

“To design a highly available web application across multiple Availability Zones, I ensure there is no single point of failure at any layer.

At the entry level, I use Route53 to route user traffic to the application.

In the network layer, I create a VPC with public and private subnets across at least two or three Availability Zones.

I deploy an Application Load Balancer in public subnets, which distributes incoming traffic across application services running in private subnets across multiple AZs.

For the application layer, I use Amazon EKS with worker nodes distributed across multiple AZs.
I configure multiple pod replicas along with HPA for pod scaling and Cluster Autoscaler for node scaling.

Kubernetes ensures self-healing using liveness and readiness probes, and ALB performs health checks to route traffic only to healthy instances.

For the database layer, I use Amazon RDS with Multi-AZ enabled, which provides automatic failover using a standby replica without changing application endpoints.

For static storage, I use Amazon S3, which is inherently highly available across AZs.

For monitoring, I use CloudWatch and Kubernetes metrics to track application and infrastructure health.

The overall flow is:

**User → Route53 → ALB → EKS (Multi-AZ Pods) → RDS (Multi-AZ) → Monitoring**

This ensures high availability, scalability, and automatic failover even if an Availability Zone goes down.”


**🔥 Pro-level Add-on**

👉 “I always ensure there is no single point of failure across network, compute, and database layers.”

## Storage (S3,EBS,EFS)

<details>
<summary><b>What are different storage services in AWS and how do you choose them?</b></summary>

<br>

AWS provides different storage services to store **files, block data, and shared data** based on use case.

Main types:

* Object Storage → S3 → Buckets (Objects as Files)
* Block Storage → EBS → Attach disk (single instance)
* File Storage → EFS → Share files (multiple instances)

We use different storage types because:

* Different applications have different needs
* Performance, scalability, and access patterns vary
* Cost optimization

**Simple Comparison (VERY IMPORTANT)**

| Feature  | S3         | EBS       | EFS                |
| -------- | ---------- | --------- | ------------------ |
| Type     | Object     | Block     | File               |
| Access   | HTTP/API   | EC2 only  | Multiple instances |
| AZ       | Multi-AZ   | Single AZ | Multi-AZ           |
| Use case | Files/logs | OS/DB     | Shared storage     |

<summary><b>Explain Amazon S3 in your project (final interview flow)</b></summary>

### 🔥 Final Answer (Say This)

“Amazon S3 is an object storage service used to store unstructured data like logs, backups, and user uploads.

We use S3 because it is highly durable, scalable, and automatically stores data across multiple availability zones, ensuring high availability without manual effort.

In terms of working, S3 stores data in the form of buckets and objects, where a bucket acts as a container and objects are the actual files.

In my project, we used S3 primarily for storing application logs and user-uploaded files. The application interacts with S3 using APIs to upload and retrieve data.

We also implemented lifecycle policies to move older logs to Glacier for cost optimization and enabled versioning to protect against accidental deletions.

For security, we used IAM roles and bucket policies, and ensured public access was blocked unless explicitly required.

Overall flow is:

**Application → S3 Bucket → Store/Retrieve Objects → Lifecycle → Archive**

This helped us achieve scalable, highly available, and cost-optimized storage.”

“In one use case, logs older than 30 days were automatically moved to Glacier, reducing storage cost significantly.”

**Pro-Level Add-on (VERY IMPORTANT)**

👉 “Since S3 is multi-AZ by default, we didn’t need to handle replication manually for high availability.”

# download file
aws s3 cp s3://my-first-bucket/file.txt .

<summary><b>S3 Bucket Policy vs IAM Role — what is the difference?</b></summary>

### 🔥 Perfect Answer (Say This)

<br>

“IAM roles and S3 bucket policies are both used to control access, but they work at different levels.

IAM roles are identity-based policies, which define what actions a user, service, or resource can perform on AWS resources.

S3 bucket policies are resource-based policies, which define who can access a specific S3 bucket and what actions they can perform.

In simple terms, IAM controls ‘who can do what’, and bucket policies control ‘who can access this resource’.

In my project, we used IAM roles for applications running on EKS to access S3 securely using IRSA, and we used bucket policies to restrict access to specific roles and block public access.

So both work together to provide secure and fine-grained access control.”

### 🧠 Simple Difference (VERY IMPORTANT)

| Feature     | IAM Role             | Bucket Policy         |
| ----------- | -------------------- | --------------------- |
| Type        | Identity-based       | Resource-based        |
| Attached to | User/Role/Service    | S3 Bucket             |
| Controls    | What actions allowed | Who can access bucket |
| Scope       | Multiple services    | Only S3               |

**Real Example (Say This)**

“In our project, pods used IAM roles via IRSA to access S3, and bucket policy ensured only that role could access the bucket, preventing any public or unauthorized access.”

<summary><b>Explain Amazon EBS in your project (final interview flow)</b></summary>

<br>

### 🔥 Final Answer (Say This)

“Amazon EBS is a block storage service used to provide persistent storage to EC2 instances.

We use EBS when we need storage for operating systems, databases, or applications that require low latency and high performance.

In terms of working, EBS volumes are attached to EC2 instances like a hard disk, and the data persists even if the instance is stopped or restarted.

In my project, we used EBS volumes attached to EKS worker nodes to store application data and logs.

We also used EBS-backed Persistent Volumes in Kubernetes for stateful applications.

We configured different volume types like gp3 for general workloads and io1 for high IOPS requirements depending on performance needs.

We also took regular snapshots of EBS volumes and stored them in S3 for backup and disaster recovery.

Overall flow is:

**Application → EC2/EKS Node → EBS Volume → Read/Write Data**

This helped us achieve low latency, high performance, and reliable storage for our applications.”

**Real Example (Say This)**

“In one use case, we used EBS Persistent Volumes for a stateful application running in EKS, ensuring that even if the pod restarted, the data was not lost.”

**Key Features (Mention if asked)**

* Persistent storage
* Low latency (attached to EC2)
* Snapshot backup (stored in S3)
* Multiple volume types (gp3, io1, st1, sc1)

**Pro-Level Add-on (VERY IMPORTANT)**

👉 “Since EBS is limited to a single AZ, we ensured high availability at the application level using replication or Kubernetes rescheduling.”

**Q: Can EBS be attached to multiple instances?**
👉 No (except multi-attach for specific cases)

**Q: How do you take backup?**
👉 Snapshots (stored in S3)

<summary><b>Explain Amazon EFS in your project (final interview flow)</b></summary>

<br>

### 🔥 Final Answer (Say This)

“Amazon EFS is a fully managed file storage service that provides a shared file system which can be accessed by multiple EC2 instances or Kubernetes pods simultaneously.

We use EFS when multiple services or pods need access to the same data at the same time.

In terms of working, EFS uses NFS protocol and can be mounted on multiple instances across different availability zones, providing high availability by default.

In my project, we used EFS in our EKS cluster for shared storage where multiple pods needed access to the same files, such as for content storage and shared configurations.

Since EFS is multi-AZ, it ensured high availability without additional configuration.

Overall flow is:

**Multiple Pods/Instances → EFS Mount → Shared Read/Write Data**

This helped us achieve shared access, scalability, and high availability for stateful applications.”

**Real Example (Say This)**

“In one use case, we had multiple pods that needed to read and write common files.
We used EFS so all pods could access the same storage without data duplication.”

**Key Features (Mention if asked)**

* Shared file system (NFS-based)
* Multi-AZ by default
* Scales automatically
* Accessible from multiple instances

**Pro-Level Add-on (VERY IMPORTANT)**

👉 “EFS is best suited for shared workloads, but it has higher latency compared to EBS, so we use it only when shared access is required.”

**Follow-up Questions (Be Ready)**

👉 EBS cannot be mounted to multiple instances (generally)
