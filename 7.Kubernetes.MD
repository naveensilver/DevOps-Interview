<br>

#### 🔥 Perfect Answer (Production-Level)

“Yeah, I’ve faced similar issues in production. I usually follow a structured layer-by-layer debugging approach to quickly isolate the problem.

First, I validate whether the Kubernetes Service is correctly routing traffic to pods using commands like `kubectl get svc`, `kubectl describe svc`, and `kubectl get endpoints`. If endpoints are empty, it usually indicates a selector mismatch.

If the Service looks fine, I move to the Ingress or Load Balancer layer. I check Kubernetes ingress resources and then verify AWS ALB target group health, listeners, and routing rules. Common issues here include unhealthy targets or incorrect path routing.

Next, I validate at the pod level by exec’ing into the pod and testing the application locally using curl. Even if pods are running, the application inside might not be responding.

Then I check network-related configurations such as Kubernetes NetworkPolicies, AWS Security Groups, and NACLs to ensure traffic is not being blocked.

If the application is accessed via domain, I verify DNS resolution using Route53 and tools like nslookup or dig.

I also check logs and events using `kubectl logs`, `kubectl describe pod`, and `kubectl get events` to identify any hidden errors.

One important step is checking recent changes like new deployments, ConfigMap or Secret updates, or Ingress modifications, as most issues are caused by recent changes.

Finally, I perform a quick isolation test using port-forwarding. If the application works via port-forward, then the issue is likely in the external layers like Ingress or Load Balancer.

So overall, I debug from Service → Ingress → Pod → Network → DNS → Logs, and isolate the issue layer by layer.”

<summary><b>How do you design multi-region failover for EKS? Explain traffic routing, data consistency, and failover trigger.</b></summary>

<br>

#### 🔥 Perfect Answer (Production-Level)

“Yeah, for high availability, I design multi-region architecture using active-passive or active-active depending on business criticality. Let me explain a typical active-passive setup which I’ve worked on.

First, I provision two EKS clusters in different regions using Terraform — one acts as the primary region handling active traffic, and the other is a secondary standby region. Both clusters are kept identical in terms of application deployments and configurations using Helm charts or Kubernetes manifests.

For traffic routing, I use Route53 with failover routing policy. The primary record points to the ALB in the active region, and the secondary record points to the ALB in the standby region.

I configure health checks on the primary ALB endpoint. If the health check fails — due to ALB failure or application not responding — Route53 automatically shifts traffic to the secondary region.

For data consistency, it depends on the application type. For stateless applications, it’s straightforward since no data sync is required. But for stateful applications, I use cross-region replication strategies such as RDS read replicas, S3 cross-region replication, or DynamoDB global tables to ensure the standby region has up-to-date data.

I also ensure secrets and configurations are synchronized across both regions using services like AWS Secrets Manager replication and consistent IAM roles and policies.

Failover is primarily automatic using Route53 health checks, but I also keep a manual failover option for controlled switching during critical situations.

For monitoring, I use tools like Prometheus, Grafana, or CloudWatch to continuously monitor both regions and trigger alerts in case of failures or failover events.

Overall, this setup ensures high availability with seamless traffic routing, consistent data replication, and automated failover handling.”

<summary><b>How do you secure a Kubernetes cluster in production?</b></summary>

<br>

#### 🔥 Perfect Answer (Production-Level)

“In production, I secure the Kubernetes cluster using a layered security approach covering access control, network, workload, and runtime security.

First, for access control, I use RBAC to enforce least privilege access. Users and service accounts are given only the permissions they need. For cloud environments like EKS, I integrate IAM with Kubernetes using IRSA so pods can securely access AWS services without static credentials.

Second, I secure the cluster at the network level using Network Policies to restrict pod-to-pod communication. Only required services can talk to each other, which reduces the attack surface.

Third, I ensure secrets are not hardcoded. Instead, I use external secret managers like AWS Secrets Manager or HashiCorp Vault and inject secrets securely into pods using External Secrets or CSI drivers.

Fourth, I secure workloads by enforcing policies. I use Pod Security Standards to restrict privileged containers, disable root access, and enforce read-only file systems. I also scan container images using tools like Trivy before deployment to ensure no vulnerabilities are introduced.

Fifth, I secure the cluster at runtime by enabling logging and monitoring. I use tools like Prometheus, Grafana, and CloudWatch for monitoring, and enable audit logs to track suspicious activity.

Finally, I ensure cluster components are regularly updated and patched, and I restrict API server access using private endpoints and security groups.

Overall, I follow a defense-in-depth approach to ensure the cluster is secure at every layer.”

---

👉 6-layer security model:

1. Access Control

* RBAC
* IAM + IRSA
* Least privilege

2. Network Security

* Network Policies
* Restrict pod communication

3. Secrets Management

* AWS Secrets Manager / Vault
* No hardcoding

4. Workload Security

* Pod Security Standards
* No root / no privileged containers
* Image scanning (Trivy)

5. Monitoring & Logging

* Prometheus / Grafana
* CloudWatch / Audit logs

6. Cluster Hardening

* Private API server
* Regular patching
* Secure etcd
