<summary><b>What is Terraform Drift and how do you handle it?</b></summary>

<br>

Yeah, Terraform drift is a situation where the actual infrastructure in the cloud differs from what is defined in the Terraform code or stored in the Terraform state file.

This usually happens when someone makes manual changes directly in AWS, like modifying an EC2 instance type or changing a security group, without updating the Terraform code.

**Flow:**

Terraform Code → Apply → Infra Created → Manual Change → Drift

In production, we detect drift by running `terraform plan`, either manually or through a scheduled CI/CD pipeline. If there’s any difference between the state file and actual infrastructure, Terraform shows it in the plan output.

Once drift is detected, we don’t directly apply changes in production. First, we analyze whether the change was intentional or not. If it was a valid change, like emergency scaling, we update the Terraform code to match the current state. If it was an unauthorized or incorrect change, we revert it using Terraform apply through our CI/CD pipeline with proper approvals.

For example, in one case, someone manually opened port 22 in a production security group. Our drift detection pipeline caught it, and we reverted the change through Terraform after review to maintain security compliance.

So overall, we treat drift carefully in production — we always review, decide whether to accept or revert, and ensure all changes go through Terraform and CI/CD to keep infrastructure consistent and controlled.

---

**Quick Definition (Short Answer)**

Drift is when actual infrastructure differs from Terraform state, usually due to manual changes. We detect it using `terraform plan` and fix it through CI/CD after review.

**Key Points to Remember**

- Drift = mismatch between state and real infra
- Caused by manual changes
- Detected using `terraform plan`
- Never auto-fix in production
- Always review before applying changes

**Real Production Scenario**

- Manual change: Port 22 opened in security group
- Drift detected via CI/CD pipeline
- Reviewed and reverted using Terraform
- Security compliance restored

**Strong Closing Line**

We always ensure that Terraform remains the single source of truth, and all infrastructure changes go through CI/CD with proper approvals.

<summary><b>What is Terraform State File and why is it important?</b></summary>

<br>

Yeah, Terraform state file is basically a file where Terraform keeps track of all the infrastructure resources it has created and manages.

The main problem it solves is tracking. Without the state file, Terraform won’t know what resources already exist, what needs to be updated, or what should be deleted.

**Flow:**

Terraform Code → Apply → Resources Created → State File Updated → Plan compares State vs Real Infra

In production, whenever we run `terraform apply`, Terraform updates the state file with details like resource IDs, configurations, and dependencies. During `terraform plan`, it compares the state file with actual infrastructure to determine what changes are required.

We usually don’t store the state file locally. In real environments, we store it remotely in S3 with DynamoDB locking so that multiple team members can work safely without conflicts and to avoid state corruption.

For example, in our project, when we created EC2 instances and VPC using Terraform, all resource details like instance IDs and subnet IDs were stored in the state file. This helped Terraform understand what already exists and avoid recreating resources.

So overall, the state file is very important because it acts as a source of truth, enabling Terraform to track infrastructure, detect changes, and safely manage resources in a team environment.

---

**Quick Definition (Short Answer)**

Terraform state file is a file that stores the current state of infrastructure managed by Terraform, helping it track resources and perform updates efficiently.

**Key Points to Remember**

- Stores mapping between Terraform code and real infrastructure
- Maintains resource metadata (IDs, configs, dependencies)
- Helps in incremental changes (no full recreation)
- Used by `plan` to detect changes
- Enables team collaboration via remote backend
- Supports state locking (DynamoDB)
- Critical for drift detection
- Should never be stored locally in production

**Real Production Scenario**

- Created EC2 and VPC using Terraform
- Resource IDs stored in state file
- Team used remote backend (S3 + DynamoDB)
- Prevented conflicts and duplicate resource creation

**Strong Closing Line**

We treat the Terraform state file as the backbone of our infrastructure management, and we always store it securely in a remote backend to ensure consistency and safe team collaboration.

<summary><b>What happens if Terraform state file is lost?</b></summary>

<br>

Yeah, if the Terraform state file is lost, Terraform basically loses track of all the infrastructure it was managing.

The main issue here is that Terraform no longer knows what resources already exist in the cloud. So when we run `terraform plan` or `apply`, it may think everything needs to be created from scratch.

**Flow:**

State File Lost → Terraform has no tracking → Plan shows all resources as new → Risk of duplicate resource creation

In production, this is a critical situation because Terraform might try to recreate existing resources, which can lead to duplication, conflicts, or even downtime.

To handle this, we usually recover the state file from backup. That’s why in production we always store state in a remote backend like S3 with versioning enabled, so we can restore previous versions.

If backup is not available, we may need to manually rebuild the state using `terraform import` to map existing infrastructure back into Terraform.

For example, if an EC2 instance is already running but the state file is lost, Terraform may try to create another instance. So we import the existing instance into Terraform state to avoid duplication.

So overall, losing the state file is a critical risk, and that’s why we always use remote backend with versioning and proper backup strategies.

---

**Quick Definition (Short Answer)**

If the state file is lost, Terraform cannot track existing resources and may try to recreate them, leading to duplication or conflicts.

**Key Points to Remember**

- Terraform loses track of infrastructure
- Plan shows all resources as new
- Risk of duplicate resource creation
- Always use remote backend with versioning
- Recovery via S3 version restore or backup
- Use `terraform import` if state cannot be recovered

**Real Production Scenario**

- State file accidentally deleted locally
- Restored previous version from S3 backend
- Avoided duplicate infrastructure creation

**Strong Closing Line**

We always ensure state files are stored in remote backends with versioning enabled, so even if something goes wrong, we can recover safely without impacting production.
